## Topic 设计

| Topic 名称 | 分区数 | 副本数 | 说明 |
|-----------|-------|-------|------|
| `orders.taobao` | 6 | 3 | 淘宝/天猫订单 |
| `orders.jd` | 6 | 3 | 京东订单 |
| `orders.douyin` | 6 | 3 | 抖音订单 |
| `orders.kuaishou` | 6 | 3 | 快手订单 |
| `orders.all` | 12 | 3 | 合并订单（可选） |
| `orders.dlq` | 3 | 3 | 死信队列 |

## 消费者组设计

| 消费者组 | 订阅Topic | 说明 |
|---------|----------|------|
| `erp-order-consumer` | orders.* | ERP订单处理 |
| `data-analytics-consumer` | orders.* | 数据分析 |
| `notification-consumer` | orders.* | 通知服务 |

## 数据流

```
1. 电商平台产生订单
2. 平台推送订单到我们的系统 (Webhook/API)
3. 订单适配器转换为统一格式
4. 生产者发送到对应Kafka Topic
5. ERP消费者消费消息
6. 数据入库并触发业务逻辑
```

## 容错与重试机制

```

└─────────────┘     └─────────────┘     └──────┬──────┘
                                               │
                                               │ 重试3次仍失败
                                               ▼
                                        ┌─────────────┐
                                        │  死信队列    │
                                        │  (DLQ)      │
                                        └─────────────┘
```
